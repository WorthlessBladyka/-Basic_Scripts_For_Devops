# Общее
 - ==Inventory== - файл, хранящий информация о целевых системах. По умолчанию Ansible использует тот, что лежит по адресу /etc/ansible/hosts.
 !Ansible может одновременно работать с одной или несколькими системами. В процессе работы он устанавливает подключение к этим системам. Это достигается через SSH в случае Linux или PowerShell с Windows Remote Management, если это Windows.

![[Снимок экрана 2025-11-09 в 1.10.08 AM.png]]
- INI-формат
- ==Inventory== - Это просто строки с записанными в них именами серверов, идущими друг за другом, по строке на сервер
- **Группы серверов** применяются для совместной обработки Ansible. Для этого нужно написать название группы в квадратных скобках, а ниже указать список серверов
- **Все** хосты, независимо от вхождения в какую-либо группу, по умолчанию входят в группу **all**
# Параметры
### алиас
Например, есть список серверов с цифрами от 1 до 4. 
Но было бы удобно, чтобы для работы с ними Ansible использовал алиас, вроде app для сервера приложений, db для сервера базы данных и т.д. 
Можно этого добиться добавив такой алиас для каждого сервера в начало его линии, а далее указав имя или адрес, на который этот псевдоним будет ссылаться с помощью параметра `ansible_host`. 
**Ansible_host** это параметр inventory, который используется для указания **полного доменного имени** (FQDN) или IP- адреса сервера.
### ansible_connection
!Параметр ansible_connection говорит Ansible, как ему пробовать подключаться к этому хосту
!Можно также указать, что Ansible следует работать с localhost, установив параметр в `local`
- local
- ssh
- winrm
### ansible_port
!Ansible_port определяет на какой порт подключаться
!По умолчанию это порт 22 для SSH
### ansible_user
!Ansible_user отвечает за пользователя, под чьим именем будет установлено удаленное подключение
!По умолчанию `root` для Linux
### ansible_ssh_pass and ansible_password
!Ansible_password устанавливает пароль для windows
!Ansible_ssh_pass устанавливает пароль SSH для Linux
!!!В этом случае, пароль будет храниться в обычном текстовом формате, что крайне ==не секьюрно==
Лучшая практика - использовать беспарольную межсерверную аутентификацию с помощью SSH ключей в своем продакшене или корпорейте.
# Agentless and + -
Ansible делает свободным от использования агента в управляемой им системе. Agentless значит, что не нужно устанавливать никаких дополнительных приложений на целевых машинах, чтобы Ansible смог ими управлять.  Одним из основных недостатков большинства других инструментов автоматизации и управления является то, что необходимо настроить агент в целевых системах, прежде чем мы сможем начать полноценно управлять этим хостом. Как и всегда, в отсутствии агента есть и недостатки, а именно неосведомленность о состоянии управляемого узла до подключения к нему, что заставляет более тщательно составлять и тестировать playbooks.
